## 1. 概述

### 1.1 背景

为了精确分析用户注册行为，后端在注册接口新增了 `register_source` 字段，用于区分：

- **主动注册**：用户主动点击「注册」按钮完成注册
- **被动注册**：用户想操作页面某个功能/内容，因未登录触发 401，前端弹出注册弹窗后完成注册

### 1.2 前端需要做什么

在调用注册接口时，根据用户触发注册的场景，传入对应的 `register_source` 值。

**工作量评估：改动极小，仅需在注册请求体中多传一个字段。**

## 2. 核心概念

### 2.1 注册行为分类

```
┌─────────────────────────────────────────────────────────┐
│                    用户注册行为                           │
├──────────────────────┬──────────────────────────────────┤
│      主动注册         │           被动注册                │
│  (用户有明确注册意图)   │  (用户想操作内容，被迫注册)        │
├──────────────────────┼──────────────────────────────────┤
│                      │                                  │
│  点击「注册」按钮      │  点击某功能 → 401 → 弹窗注册      │
│  register_source:    │  register_source:                │
│    "direct"          │    "auth_redirect"               │
│                      │                                  │
│  点击「Google 登录」   │  应用内引导弹窗 → 注册             │
│  (后端自动标记，       │  register_source:                │
│   前端无需处理)        │    "in_app_prompt"               │
│                      │                                  │
└──────────────────────┴──────────────────────────────────┘
```

### 2.2 值定义

| register_source | 含义 | 触发场景 | 分类 |
|----------------|------|---------|------|
| `"direct"` | 主动点击注册按钮 | 用户在注册页/登录页主动点击「注册」 | 主动注册 |
| `"auth_redirect"` | 401 弹窗后注册 | 用户操作内容 → 401 → 弹出注册弹窗 → 注册 | 被动注册 |
| `"in_app_prompt"` | 应用内引导后注册 | 功能引导弹窗/横幅提示注册 → 用户点击注册 | 被动注册 |
| `"google_oauth"` | Google 登录 | 用户点击 Google 登录按钮 | **后端自动标记** |
| 不传 / `null` | 默认处理 | 未传时后端默认为 `"direct"` | 主动注册 |

## 3. API 变更

### 3.1 注册接口

```
POST /api/user/register
```

#### 请求体 (变更部分)

```jsonc
{
  "email": "user@example.com",
  "password": "123456",
  "register_source": "direct"    // [新增字段] 可选，见下方值说明
}
```

#### register_source 字段规格

| 属性 | 值 |
|------|-----|
| 字段名 | `register_source` |
| 类型 | `string \| null` |
| 是否必填 | 否（不传默认 `"direct"`） |
| 合法值 | `"direct"`, `"auth_redirect"`, `"in_app_prompt"`, `"channel_link"` |
| 非法值处理 | 返回 422 Validation Error |

#### 响应 (无变化)

```json
{
  "code": 200,
  "message": "User registration successful",
  "data": {
    "id": 1,
    "email": "user@example.com",
    "is_active": true,
    "is_superuser": false,
    "access_token": "eyJ...",
    "token_type": "bearer"
  }
}
```

### 3.2 非法值的错误响应

如果传入不在合法值列表中的字符串：

```json
// 请求
{ "email": "a@b.com", "password": "123456", "register_source": "xxx" }

// 响应 422
{
  "detail": [
    {
      "type": "value_error",
      "loc": ["body", "register_source"],
      "msg": "Value error, Invalid register_source: xxx"
    }
  ]
}
```

## 4. 前端实现指南

### 4.1 总体方案

前端需要在调用注册 API 时，根据触发注册的上下文传入正确的 `register_source`。

**核心思路：在触发注册的时刻记录来源，而非在注册表单提交时判断。**

### 4.2 实现方案：注册上下文状态

#### 步骤 1：定义注册来源常量

```typescript
// constants/registration.ts

/** 注册来源 — 与后端 RegistrationSource 一一对应 */
export const REGISTER_SOURCE = {
  /** 主动注册：用户主动点击注册按钮 */
  DIRECT: 'direct',
  /** 被动注册：用户操作内容时 401，弹窗后注册 */
  AUTH_REDIRECT: 'auth_redirect',
  /** 被动注册：应用内引导弹窗提示后注册 */
  IN_APP_PROMPT: 'in_app_prompt',
} as const;

export type RegisterSource = typeof REGISTER_SOURCE[keyof typeof REGISTER_SOURCE];
```

#### 步骤 2：注册上下文状态管理

```typescript
// stores/auth-store.ts (以 Zustand 为例，其他状态库同理)

import { REGISTER_SOURCE, RegisterSource } from '@/constants/registration';

interface AuthStore {
  /** 当前注册来源上下文 */
  registerSource: RegisterSource;
  /** 设置注册来源 */
  setRegisterSource: (source: RegisterSource) => void;
  /** 重置为默认值 */
  resetRegisterSource: () => void;
}

export const useAuthStore = create<AuthStore>((set) => ({
  registerSource: REGISTER_SOURCE.DIRECT,
  setRegisterSource: (source) => set({ registerSource: source }),
  resetRegisterSource: () => set({ registerSource: REGISTER_SOURCE.DIRECT }),
}));
```

#### 步骤 3：在 401 拦截器中设置来源

```typescript
// lib/api-client.ts (Axios 拦截器示例)

import { useAuthStore } from '@/stores/auth-store';
import { REGISTER_SOURCE } from '@/constants/registration';

axios.interceptors.response.use(
  (response) => response,
  (error) => {
    if (error.response?.status === 401) {
      // ★ 关键：标记本次注册来源为 "被动注册（401 弹窗）"
      useAuthStore.getState().setRegisterSource(REGISTER_SOURCE.AUTH_REDIRECT);

      // 弹出注册/登录弹窗
      openAuthModal();
    }
    return Promise.reject(error);
  }
);
```

#### 步骤 4：在注册请求中携带来源

```typescript
// api/auth.ts

import { useAuthStore } from '@/stores/auth-store';

export async function registerUser(email: string, password: string) {
  const { registerSource, resetRegisterSource } = useAuthStore.getState();

  const response = await axios.post('/api/user/register', {
    email,
    password,
    register_source: registerSource,   // ★ 携带注册来源
  });

  // 注册成功后重置来源
  resetRegisterSource();

  return response.data;
}
```

#### 步骤 5：在主动注册入口设置来源

```tsx
// components/RegisterButton.tsx

import { useAuthStore } from '@/stores/auth-store';
import { REGISTER_SOURCE } from '@/constants/registration';

function RegisterButton() {
  const setRegisterSource = useAuthStore((s) => s.setRegisterSource);

  const handleClick = () => {
    // ★ 用户主动点击注册，标记为主动注册
    setRegisterSource(REGISTER_SOURCE.DIRECT);
    openAuthModal();
  };

  return <button onClick={handleClick}>注册</button>;
}
```

}

## 5. 场景触发规则

### 5.1 前端判断规则速查表

| 用户行为 | register_source | 触发时机 |
|---------|----------------|---------|
| 点击顶部导航栏「注册」按钮 | `"direct"` | 按钮 onClick |
| 点击登录页的「没有账号？注册」链接 | `"direct"` | 链接 onClick |
| 点击首页 Hero 区域的注册 CTA | `"direct"` | 按钮 onClick |
| 点击任意内容 → 401 → 弹窗注册 | `"auth_redirect"` | 401 拦截器 |
| 点击「查看更多」→ 401 → 弹窗注册 | `"auth_redirect"` | 401 拦截器 |
| 点击聊天输入框 → 401 → 弹窗注册 | `"auth_redirect"` | 401 拦截器 |
| 功能引导弹窗中的「立即注册」 | `"in_app_prompt"` | 引导弹窗按钮 onClick |
| 底部横幅「注册免费获取 XX」 | `"in_app_prompt"` | 横幅按钮 onClick |

### 5.2 优先级规则

当多个场景叠加时（如渠道链接进入 → 点击内容 → 401 → 注册），以**最后一次 `setRegisterSource` 的值为准**：

```
优先级（后设覆盖前设）：
  401 拦截器设置 > 按钮点击设置 > 默认值
```

这是正确的行为，因为 401 弹窗注册一定是在用户点击内容之后触发的，时间上更晚、更能反映真实注册动机。

## 6. Google OAuth 说明

### 6.1 前端无需传 register_source

Google OAuth 的注册来源由**后端自动标记为 `"google_oauth"`**。

前端只需正常跳转到 Google 登录即可，无需传递任何额外参数：

```typescript
// Google OAuth 登录 — 无需修改
function handleGoogleLogin() {
  window.location.href = '/api/auth/google/login';
  // 后端自动处理：新用户 → register_source = "google_oauth"
  //              老用户 → 正常登录，不影响
}
```

### 6.2 Google 登录后 401 场景

如果用户先触发了 401（`setRegisterSource("auth_redirect")`），但选择用 Google 登录：
- **不影响**：后端对 Google OAuth 始终标记为 `"google_oauth"`，不读取前端传的值
- 前端的 `registerSource` 状态会在下次注册/重置时清除

## 8. FAQ

### Q1: 如果前端不传 register_source 会怎样？

后端会默认设为 `"direct"`（主动注册）。所以最坏情况下，不传也不会报错，只是所有注册都会被统计为主动注册。

### Q2: 传了非法值会怎样？

后端会返回 `422 Validation Error`，注册请求被拒绝。合法值只有 5 个：`direct`, `auth_redirect`, `in_app_prompt`, `google_oauth`, `channel_link`。

### Q3: 用户先触发 401 弹窗，但关闭弹窗后自己去注册页注册了？

如果用户关闭弹窗后手动导航到注册页并点击「注册」按钮，此时应该触发按钮的 `setRegisterSource("direct")`，覆盖之前 401 设的 `"auth_redirect"`。这是正确的——因为用户最终是主动去注册的。

### Q4: Google OAuth 用户可以被统计为被动注册吗？

不可以。Google OAuth 流程经过了 Google 服务器重定向，前端的 `registerSource` 状态无法传递到后端回调。后端对 Google OAuth 一律标记为 `"google_oauth"`（主动注册）。

### Q5: 已经在线上运行的老版本前端怎么办？

完全兼容。老版本不传 `register_source`，后端默认为 `"direct"`。但统计准确性会降低（所有注册都被计为主动注册），建议尽快升级。

### Q6: 渠道链接用户的注册来源是什么？

渠道归因由后端的渠道追踪系统（指纹匹配 + `ChannelConversionModel`）独立处理，与 `register_source` 无关。渠道用户的 `register_source` 仍然取决于注册触发方式：
- 通过渠道链接进入 → 主动点击注册 → `"direct"`
- 通过渠道链接进入 → 点击内容 → 401 → 弹窗注册 → `"auth_redirect"`
